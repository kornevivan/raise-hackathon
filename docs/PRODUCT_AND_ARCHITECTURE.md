# Covenant Sentinel — продукт и архитектура агента

Документ описывает, что агент умеет, как устроено взаимодействие с пользователем, и как
он решает задачу: что детерминировано, что отдано на суждение модели, и на какую
специфику решение заточено.

---

## 1. Продуктовые характеристики

### Кого обслуживает
**Кредитный портфельный аналитик / специалист loan servicing** в банке или прямом
кредиторе. Его боль: ковенантные проверки квартальные, ручные и медленные — перечитать
200-страничный кредитный договор ради точного определения «Consolidated EBITDA»,
пересобрать коэффициент в Excel, найти в реестре причину изменения. Дни работы на одного
заёмщика. А поправки к договору незаметно меняют определения — наивный расчёт даёт
неверный ответ, и ложные срабатывания стоят дорого (waiver’ы, испорченные отношения).

### Что агент умеет
1. **Читает реальные документы** — кредитные договоры, поправки, финотчётность,
   compliance-сертификаты, включая **таблицы и сканы** (через VultronRetriever).
2. **Сам пересчитывает ковенантные коэффициенты** детерминированным калькулятором — он
   никогда не считает «в уме».
3. **Мотивированно до-извлекает** — если определение EBITDA ссылается на поправку,
   агент замечает это и идёт за поправкой второй раз (это и есть ключевой «multi-step»).
4. **Копает причину** в реестре транзакций (что именно уронило EBITDA / выручку).
5. **Готовит decision-ready меморандум**: ситуация, расчётный след, причина,
   рекомендация + оценка уверенности. **Каждое фактическое утверждение снабжено
   кликабельной цитатой** на конкретный пункт/ячейку исходной страницы.
6. **Оставляет решение человеку**: Approve / Escalate / Send back.

### Фирменная способность (демо-момент)
Наивный расчёт даёт **3.55×** — пробой ковенанта (лимит 3.50×). Агент замечает, что
определение EBITDA изменено **Amendment No. 1**, до-извлекает поправку, применяет
разрешённый add-back по разовым затратам на поглощение и пересчитывает — **3.42×, пробоя
нет**. Он поймал ложное срабатывание, потому что прочитал документы как человек.

### Два режима работы
- **Sample portfolio** — портфель из 3 заёмщиков с готовыми документами (мгновенно, для
  демонстрации разных исходов: ложный пробой / реальный пробой / всё чисто).
- **Upload your documents** — прод-режим: пользователь загружает свои PDF (договор +
  отчётность + при наличии поправку), и агент **сам определяет ковенант и порог**,
  извлекает числа из документов и проводит анализ. Никакой настройки.

## 2. Взаимодействие с пользователем (UX)

**Двухпанельный интерфейс, всё в реальном времени (SSE-стрим):**

- **Слева — живой трейс агента**: plan → retrieve → tools → gap-check → verify → memo.
  Каждый шаг появляется по мере выполнения с бейджами: какой **тир** (Flash/Core/Prime),
  какая **модель**, **латентность**, и метка источника — `Vultr` (живой вызов) против
  `deterministic` (детерминированный инструмент/фолбэк). Ретривы показывают миниатюры
  найденных страниц и **«почему»** сделан именно этот (и почему понадобился второй).
- **Справа — меморандум / просмотрщик документа**:
  - Плашка коэффициента с эффектом переворота **3.55× (красный, зачёркнут) → 3.42×
    (зелёный)** — исход виден мгновенно.
  - Меморандум с разделами и **кликабельными маркерами `[n]`**. Клик → правая панель
    открывает исходную страницу и **подсвечивает точный пункт/ячейку** (в т.ч. на
    отсканированной или загруженной пользователем странице).
  - Бейдж рекомендации + оценка уверенности, список источников.
  - Кнопки **Approve / Escalate / Send back** — человек принимает решение.

**Режим загрузки**: drag-and-drop PDF → «Analyze». Есть встроенные sample-PDF для проверки
в один клик. Если в документах **нет нужных цифр**, агент честно выдаёт вердикт
**«Insufficient data»** с перечнем «что нашёл / чего не хватает» — не падает и не выдумывает.

## 3. Архитектура агентского решения

### Принцип: фиксированный скелет в коде, суждение модели — только в узких точках
Оркестрация — это **явный конечный автомат в коде**. LLM вызывается только там, где нужна
интерпретация, и всегда под **строгой JSON-схемой**, с таймаутом, одной попыткой ремонта
JSON и бюджетом на прогон. Никаких свободных агентских циклов и неограниченной рекурсии.

```
Триггер: «проверить ковенант для заёмщика X / проанализировать загруженные документы»
   │
[1] PLANNER          — перечисляет проверки и какие данные нужны
   │
[2] EVIDENCE LOOP    (на проверку, ≤3 итерации, оркестрируется кодом)
   │   a. RETRIEVE   — VultronRetriever ищет страницы → цитаты {doc, page, block}
   │   b. TOOLS      — financials_query · transactions_query · ratio_calculator
   │   c. GAP-CHECK  — «определение ссылается на не применённую поправку?»
   │                   → мотивированный повторный ретрив (эскалация Flash→Prime), с логом «почему»
   │
[3] VERIFIER         — у каждого утверждения есть цитата и число совпадает с выводом инструмента
   │                   → grounded confidence = доля проверенных утверждений
[4] MEMO             — ситуация · расчётный след · причина · рекомендация, всё с цитатами
   │
[5] HUMAN-IN-THE-LOOP — Approve / Escalate / Send back
```

### Ретрив: VultronRetriever как слой поиска по документам
VultronRetriever — это **layout-aware визуальный постраничный ретривер** (семейство
ColPali / late-interaction): читает рендер страницы целиком — таблицы, графики, **сканы**.
Мы используем **три флейвора, эскалируемых по сложности**, и логируем каждое решение о
маршрутизации:
- **Flash-0.8B** — рутинный первичный поиск (дёшево, часто);
- **Core-4.5B** — стандартное извлечение доказательств;
- **Prime-8B** — сложные/неоднозначные/плотные страницы; когда gap-check требует
  аккуратного повторного поиска (например, найти поправку), ретрив эскалируется Flash→Prime.

Единица поиска — **страница**. Технически страницы индексируются в **Vultr Vector Store**,
поиск — реальный вызов `vector_store/search` с выбранной моделью VultronRetriever; результат
маппится обратно на локальные блоки страницы, чтобы цитата подсвечивалась пиксель-в-пиксель.

### Reasoning: Vultr Serverless Inference
Планировщик, gap-check, verifier и синтез меморандума работают на **чат-модели Vultr
Serverless Inference** (`deepseek-ai/DeepSeek-V4-Flash` — выбрана по стабильной низкой
задержке и надёжному JSON; сравнение альтернатив — в `backend/probe_vultr.py`). Маршрутизация
по **трём когнитивным тирам** по сложности задачи; в трейсе видно, какой тир и какая модель
отработали шаг.

### Инструменты (детерминированные, в коде)
- `ratio_calculator(numerator, denominator, addbacks[])` — точная арифметика с пошаговым
  следом (всё, что можно кликнуть и перепроверить).
- `financials_query(borrower, period, line_item)` — строки из финотчётности.
- `transactions_query(...)` — выборки из реестра транзакций (SQLite) для анализа причины.
- `extract_financials` (режим загрузки) — извлечение чисел из произвольных документов
  (regex-first + LLM-fallback).

## 4. Что детерминировано, а что — нет

Это сознательное разделение: **модель интерпретирует и формулирует, а исход и все числа
считает код.** Именно поэтому «каждое число проверяемо».

### Детерминировано (код, без участия LLM)
- **Скелет оркестрации**: последовательность шагов, границы итераций (≤3), бюджеты
  (≤14 LLM-вызовов на прогон, таймауты, watchdog).
- **Вся арифметика** — каждый коэффициент считает `ratio_calculator`, а не модель.
- **Итоговая рекомендация** — функция посчитанного коэффициента vs порога и факта
  применения add-back: `comply` / `breach` / `false_positive` / `insufficient_data`.
  LLM **не может** переопределить вердикт (в коде мы принудительно берём детерминированное
  значение поверх ответа модели).
- **Оценка уверенности** — grounded: доля утверждений, у которых есть цитата и совпадение с
  инструментом (не «на глаз»).
- **Триггер gap-check** — «определение ссылается на Amendment No. 1 и он не применён» —
  детерминированный (по тексту); модель лишь **пишет объяснение**, а не решает.
- **Извлечение чисел** — regex-first; когда стандартная строка найдена (Net Debt, EBITDA-
  компоненты), детерминированное значение **побеждает** извлечённое моделью.
- **Принцип «не выдумывать»** — если поправка упомянута, но её текст/лимит не загружен, cap
  **не фабрикуется**, add-back **не применяется**; наивный результат остаётся честно.
- **Цитаты и подсветка** — маппинг {doc, page, block, bbox} в коде.

### На суждении модели (LLM, под схемой)
- **Планирование/приоритизация**: какие проверки, приоритет риска, какие данные нужны.
- **Определение ковенанта из произвольных документов** (LLM + regex-фолбэк по порогу).
- **Объяснение gap-check** — почему до-извлечение нужно (нарратив, не вердикт).
- **Анализ причины** — текстовое объяснение, что уронило EBITDA/выручку.
- **Verifier** — оценка доли обоснованных утверждений и формулировки.
- **Проза меморандума** — тексты разделов (но числа и рекомендация зафиксированы кодом).

### Отказоустойчивость
Любой сбой отдельного LLM-вызова (таймаут, кривой JSON) **деградирует к детерминированному
результату** для этого шага — прогон не ломается на середине. Живые ответы Vultr кэшируются
по хэшу промпта (мгновенный повтор для демо). Плюс полностью **офлайн-режим** (без ключа)
воспроизводит тот же трейс/меморандум детерминированно.

## 5. На какую специфику заточено

- **Мониторинг финансовых ковенантов** — конкретно maximum total net leverage ratio:
  определения EBITDA и Total Net Debt, пороги Section 6.10, поправки, add-back’и с
  лимитами (cap). Логика причин: разовые затраты на поглощение vs реальное падение выручки.
- **Messy real-world documents** — таблицы, сноски, объединённые ячейки и **сканы**
  (скошенный, зашумлённый image-PDF): сильная сторона VultronRetriever; демо включает момент,
  где число берётся из таблицы на отсканированной странице.
- **Дисциплина малых моделей** — самый сильный ретривер 8B, быстрая чат-модель: узкие
  структурированные промпты, строгие JSON-схемы, temperature ≈ 0.1, контекст только на
  уровне страницы, one-shot repair, жёсткие бюджеты. Надёжность даёт **кодовый скелет**, а не
  «автономность» модели.
- **Доверие через проверяемость** — каждое утверждение цитируется, каждое число
  пересчитывается инструментом, финальное решение принимает человек.
- **Надёжность живого демо** — предпрогретый кэш, аккуратная деградация, честный вердикт
  «insufficient data» вместо падения или выдумывания.

## 6. Границы (честно)
- Заточено под **leverage-ковенант**; другие метрики (coverage и т.п.) — best-effort.
- В режиме загрузки числа извлекаются из документов пользователя моделью+regex; арифметика
  детерминирована, но качество извлечения зависит от разметки документа (для текстовых PDF —
  хорошо; чистые сканы без текстового слоя требуют OCR, который в этой сборке не подключён).
- Синтетические данные портфеля сгенерированы для демо (сид-скрипт), реальные компании не
  используются; подстановка реального договора из SEC EDGAR — drop-in.
